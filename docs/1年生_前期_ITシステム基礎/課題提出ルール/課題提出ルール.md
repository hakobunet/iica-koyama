# 総合制作 — 制作ルール（最低レギュレーション）

## 概要
 
以下のルールを**すべて満たした作品**を提出してください。

使う技術（GAS・localStorage など）は問いません。  
**「何ができるか」で評価します。**

---

## 必須機能要件

### 1. CRUD 操作（4つすべて必須）

HTML の画面上で以下の操作がすべてできること。

| 操作 | 内容 | 例 |
|------|------|----|
| **登録（Create）** | 新しいデータを追加できる | フォームに入力して保存ボタンを押す |
| **参照（Read）** | 登録されたデータを一覧・詳細で見られる | テーブルやカードで一覧表示する |
| **更新（Update）** | 既存のデータを編集して上書きできる | 編集ボタンを押して内容を変更する |
| **削除（Delete）** | 不要なデータを消せる | 削除ボタンを押してデータを消す |

### 2. 外部 API との通信

外部のAPIにリクエストを送り、取得したデータを画面に表示すること。

- 例：天気API・為替レートAPI・ニュースAPI など
- APIのレスポンスをそのまま表示するだけでなく、**画面の表示やシステムの動作に活用**すること

---

## 必須コード要件

技術に関わらず、コードの中に以下を含めること。

- [ ] **自作の関数**を2つ以上定義する
- [ ] **条件分岐**（`if / else`）を1つ以上書く
- [ ] **ループ**（`for` または `while`）を1つ以上書く

---

## 禁止事項

| NG | 理由 |
|----|------|
| サンプルコードをそのままコピーして提出 | 自分で書いたことにならないため |
| ユーザー操作と無関係に常に同じ画面を表示するだけ | システムとして機能していないため |
| CRUD の一部（特に更新・削除）が未実装 | 最低要件を満たしていないため |
| API を呼ぶだけで結果を何にも使っていない | 通信の意味がないため |

---

## 発表での説明要件

コマ20の発表（5分程度）では以下を**自分の言葉で**説明できること。

1. **何のために作ったか**（目的・解決する問題）
2. **どう使うか**（実際にデモして CRUD をすべて操作して見せる）
3. **外部 API をどう活用したか**
4. **コードで工夫した箇所を1つ以上**、コードを画面に表示しながら説明する

---

## 評価基準（100点満点）

| 項目 | 点数 | 基準 |
|------|------|------|
| CRUD の実装 | 40点 | 4操作それぞれ10点。動作すれば満点 |
| 外部 API の活用 | 20点 | 呼び出しのみ10点、結果をシステムに活かせていれば+10点 |
| オリジナリティ・完成度 | 20点 | 自分で考えたテーマか・エラーなく動くか |
| 発表・説明力 | 20点 | コードの仕組みを自分の言葉で説明できるか |

## 参考：「合格ライン」の例

### ✅ 最低ラインをクリアしている例

- **タスク管理ツール**：タスクの追加・一覧表示・完了に更新・削除ができ、今日の天気を API で表示
- **蔵書管理**：本を登録・一覧・編集・削除でき、ISBN から書籍情報を API で取得して自動入力
- **メモアプリ**：メモの作成・一覧・編集・削除ができ、天気 API で今日の気分を記録欄に表示

### ❌ ルールを満たしていない例

- 登録と参照だけで、更新・削除がない
- API を呼んでいるが、結果をどこにも表示・活用していない
- フォームはあるが、データが保存されない（ページを更新すると消える）

## よくある質問

**Q. GAS を使わないといけませんか？**  
A. 必須ではありません。CRUD と API の要件を満たせればどんな技術でも構いません。

**Q. データの保存先はどこでもいいですか？**  
A. GAS＋スプレッドシート・localStorage など、ページを再読み込みしてもデータが残る方法を選んでください。

**Q. チームで作ってもいいですか？**  
A. 原則1人制作です。発表でコードを自分で説明できることが条件です。
