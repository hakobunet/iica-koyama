# 授業の趣旨まとめ — 最終課題を振り返って

すでに発表は終わっていますが、今回の課題で何を意図していたのかを改めて整理します。
発表を聞いて「AIを使っていいの？」「自分で書かなくていいの？」と戸惑った人もいると思うので、その疑問への答えも含めています。

---

## 1. この授業で一番伝えたいこと

プログラミングは、説明を読んだり聞いたりするだけでは身につかず、**自分で実際に手を動かして組んでいくしか学習方法がない**と考えています。

そのための具体的な学習方法として、教材サイト **paiza** のスキルチェックを使い、

```
[ D問題 ]  ──── 数をこなす ────▶ [ C問題 ]
 基礎の構文・考え方を            少し複雑な問題に
 手で覚える                      挑戦して応用力をつける
```

という順番で進めることを勧めています。これは今回の課題が終わった後も、プログラミング力そのものを伸ばしたい人にとって変わらず王道の方法です。

---

## 2. 今回の課題の位置づけ

今回の最終課題は、上記の学習方法とは少し違う狙いを持たせています。

```
 通常の学習（プログラミング力を伸ばす）        今回の課題（システムを作る経験）
 ─────────────────────────────        ─────────────────────────────
  自分で書く                              要件を考える
     │                                       │
     ▼                                       ▼
  詰まる・調べる                          AIも使って実装する
     │                                       │
     ▼                                       ▼
  小さな問題を1問ずつ解ける               1つのシステムが動く状態まで完成させる

  目的：コードを書く力そのものをつける     目的：作り切る経験・説明できる力をつける
```

- **AIの利用を可としている**
- 重点は「正しいコードを自分の手だけで書けるか」ではなく、**「ゼロから1つのシステムを実際に作ってみる」という経験そのもの**

つまり今回は、コードを1行ずつ自分で書く練習ではなく、要件を整理し、AIも使いながら動くものを形にする、という経験を優先した課題です。

---

## 3. 発表での説明が必須な理由

課題提出ルールにある「発表での説明要件」は、今回の課題の中で最も重要な部分です。

```
AIを使ってシステムを作る
        │
        ▼
   動くようになる
        │
        ▼
 仕組みを自分で理解する   ← ここが抜けると未完成
        │
        ▼
 自分の言葉で説明できる   ← 今回ゴールとしている状態
```

AIを使って作ったシステムであっても、

- 何のために作ったか
- どう動く仕組みになっているか
- コードのどこを自分で工夫・調整したか

を**自分の言葉で説明できる**ことを求めています。AIに作らせて終わり、ではなく、出来上がったものの仕組みを理解し説明できる状態まで自分の中に落とし込んでほしい、というのがこの要件の意図です。

---

## 4. 「自分でコードを書きたい」人へ

今回AI利用可としたことで、「AIを使うのが前提なのか」と戸惑った人もいたようです。

そうではありません。**自分の力でコードを書きたい人は、書いて構いません。** 評価は「何ができるか」で見ているので、自分で書いたコードでも、AIを使ったコードでも、要件を満たし、自分で説明できれば同じように評価します。

むしろプログラミング力を本気で伸ばしたいなら、第1節で触れた paiza でのD→C問題演習のように、自分で書いて詰まって調べる、という経験は今後も避けて通れません。今回AIを許可したのは、その学習法をやめてよいという意味ではありません。

---

## 5. AIと開発の現実

今回の課題のやり方は、実際の現場の開発の進め方に近づけたものでもあります。

```
┌────────────────────────────┐
│  初期のざっくりした実装       │ ← AI が担当することが多い
└──────────────┬─────────────┘
               ▼
┌────────────────────────────┐
│  不具合の調査・細かい修正      │
│  仕組みの理解・人への説明      │ ← 人（あなた）が担当する
└────────────────────────────┘
```

現在の開発現場でも、

- 初期のざっくりした実装はAIに任せることが一般的になってきている
- 一方で、不具合の調査や細かい修正、仕組みの理解と説明は人が担う

という分担が広がっています。今回「AIで作って、最後は自分で説明する」という流れを課題にしたのは、こうした今の開発のあり方を体験してもらう意図もあります。

---

## まとめ

- プログラミング力を伸ばす王道は、自分で書いて詰まって調べること（paiza D→C）
- 今回の課題はその王道とは別軸で、「AIを使ってでも1つシステムを作り切る経験」を目的にしたもの
- AIで作っても、自分で書いても評価は同じ。大事なのは最後に自分の言葉で説明できること
- これは今の実際の開発現場の感覚に近い経験でもある
