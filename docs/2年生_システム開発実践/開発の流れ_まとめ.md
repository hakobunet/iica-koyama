## 全体像：何を作ったのか

### 完成したシステムの全体図

```
[チームメンバー]
     │ git push / Pull Request
     ▼
[GitHub リポジトリ]
     │ clone / pull
     ▼
[Laravel API サーバー]  ←──  [認証トークン（Sanctum）]
     │                              ▲
     │ SQL クエリ              ログイン時に発行
     ▼
[MySQL データベース]
     ▲
     │ JSON レスポンス
[ブラウザ / React / スマホアプリ]
```

### 5つのフェーズ

| フェーズ | テーマ | キーワード | 担当授業 |
|---------|--------|-----------|---------|
| 1 | チーム開発の準備 | ブランチ・PR・レビュー | 04/30 |
| 2 | アプリの土台を作る | Laravel・DB・マイグレーション | 05/06 |
| 3 | データの窓口を作る | REST API・JSON・ルーティング | 05/14 |
| 4 | 形と守りを整える | Resource・バリデーション・Sanctum | 05/21 |
| 5 | 動いているか確かめる | curl・ブラウザテスト | 05/28 |

## フェーズ1：チーム開発の準備

### 何をしたか

Gitのブランチ・merge・Pull Request（PR）・コードレビューを学び、
アプリをチームでGitHubに上げて全員で動かせる状態にしました。

### なぜ必要だったか

> 💡 **たとえるなら…工事現場の図面管理**
>
> 大きな建物を複数の業者が一緒に建てるとき、全員が「同じ図面の原本」に
> 直接書き込んだら混乱します。だから各業者は「自分用のコピー（ブランチ）」
> に書き込んで、完成したらまとめて照合（レビュー）してから原本に反映します。

チーム開発では複数人が同じファイルを同時に編集します。
ブランチを切ることで「メインのコードを壊さずに新機能を開発」できます。
PRは「このコードをメインに入れていいですか？」という申請で、
レビューはそのコードが正しいか・読みやすいかをチームで確認する場です。

> ⚠️ **これがないと…** 誰かのコードが誰かのコードを上書きして消えます。
> 「さっきまで動いていたのに動かなくなった」が頻発します。

### このフェーズでできるようになったこと

- `git switch -c feature/xxxx` でブランチを作って作業できる
- `git push origin feature/xxxx` でチームに共有できる
- GitHub でPRを作成してレビューを依頼できる
- レビューのコメントを受けてコードを修正できる

## フェーズ2：アプリの土台を作る

### 何をしたか

Laravel プロジェクトを作成し、データベース（`school_api`）を用意して、
`students` テーブルを定義・作成。テストデータ10件を投入してサーバーを起動しました。

### なぜ必要だったか

> 💡 **たとえるなら…飲食店の厨房づくり**
>
> お客さんに料理（APIのレスポンス）を出すには、
> まず厨房（Laravel）と食材の置き場所（データベース）が必要です。
> 厨房の設計図（マイグレーション）を先に作っておくと、
> チームの誰でも同じ環境を再現できます。

Laravelは「よく使う機能（ルーティング・DB操作・認証など）が最初から入った箱」です。
マイグレーションはデータベースの設計書で、`php artisan migrate` を実行するだけで
チーム全員が同じテーブル構造を手元に再現できます。
シーダーはテストデータの自動生成で、手作業でデータを入力する手間を省きます。

> ⚠️ **これがないと…** APIのコードを書いてもデータの保存先がないため、
> リクエストのたびにデータが消えます。

### このフェーズでできるようになったこと

- `composer create-project laravel/laravel` でプロジェクトを作れる
- `.env` でDB接続情報を設定できる
- `php artisan make:model Student -mf` でモデル・マイグレーション・ファクトリを一括生成できる
- `php artisan migrate` でテーブルを作成できる
- `php artisan db:seed` でテストデータを投入できる
- `php artisan serve` でローカルサーバーを起動できる

## フェーズ3：データの窓口を作る

### 何をしたか

REST API の概念（URL × HTTPメソッド・JSONレスポンス・ステータスコード）を学び、
`StudentController` を作成して `GET /api/students` など5つのエンドポイントを実装しました。

### なぜ必要だったか

> 💡 **たとえるなら…自動販売機のボタン**
>
> 自動販売機には「コーラ」「お茶」「水」などボタンがあり、
> 押すと対応する商品が出てきます。
> APIのエンドポイントも同じで、
> 「`GET /api/students`（全員ください）」「`DELETE /api/students/3`（3番を削除）」
> のようにURL×HTTPメソッドの組み合わせでやることが決まります。

1年生で作ったBladeアプリはHTMLページを返していました。
APIはその代わりにJSONデータだけを返します。
JSONにすることで、ReactやVue、スマホアプリなど
**異なる画面から同じデータを使い回せる**ようになります。

> ⚠️ **これがないと…** フロントエンドとサーバーが会話できません。
> ReactからAPIを叩いても「どこに送ればいいかわからない」状態になります。

### このフェーズでできるようになったこと

- REST API の URL設計（リソース名・HTTPメソッド）を理解できる
- `php artisan make:controller Api/StudentController --api` でAPIコントローラを生成できる
- `routes/api.php` にルーティングを追加できる
- `response()->json()` でJSONレスポンスを返せる
- HTTPステータスコード（200・201・404・422）の意味がわかる

## フェーズ4：形と守りを整える

### 何をしたか

`StudentResource` でレスポンスの形を統一し、バリデーションでデータの入力チェックを追加。
Laravel Sanctumで認証機能（ログイン・トークン発行・ログアウト）を実装しました。

### なぜ必要だったか

> 💡 **たとえるなら…宅配便の伝票フォーマット＋入館証**
>
> 宅配便では「送り主・宛先・品名」が決まった場所に書かれています。
> フォーマットが統一されているから、受け取る側が迷わず処理できます。
> `StudentResource` はレスポンスのフォーマットを決める「伝票の書式」です。
>
> Sanctumのトークンは「入館証」です。
> ログイン時に発行された入館証（トークン）を持っている人だけが
> 建物（API）の中に入れます。

フェーズ3でAPIは動くようになりましたが、まだ誰でもアクセスできる状態でした。
認証がないと見知らぬ人がデータを自由に取得・削除できてしまいます。
またAPIのレスポンス形式がバラバラだと、フロントエンドがデータを扱いにくくなります。

> ⚠️ **これがないと…** インターネット上の誰でもデータを書き換え・削除できます。
> パスワードやメールアドレスなどの個人情報が丸見えになる危険もあります。

### このフェーズでできるようになったこと

- `php artisan make:resource StudentResource` でレスポンス整形クラスを作れる
- `$request->validate()` でバリデーションを書ける
- `composer require laravel/sanctum` で認証パッケージを導入できる
- `POST /api/login` でトークンを取得できる
- `auth:sanctum` ミドルウェアで認証が必要なルートを保護できる

## フェーズ5：動いているか確かめる

### 何をしたか

`curl` コマンドでAPIにリクエストを送って動作確認し、
さらにブラウザで操作できるテスト画面（`/api-test`）を作って
ログイン〜生徒一覧取得〜追加〜削除の一連の流れを確認しました。

### なぜ必要だったか

> 💡 **たとえるなら…完成した自動販売機に実際にお金を入れてみる**
>
> 自動販売機を設計・製造しても、実際にお金を入れてボタンを押してみるまで
> 「本当に商品が出てくるか」はわかりません。
> APIも同じで、コードを書いただけでは正しく動くかわかりません。
> 実際にリクエストを送って、期待通りのJSONが返ってくるか確認が必要です。

`curl` は「ターミナルからHTTPリクエストを送るコマンド」です。
ブラウザを開かなくてもAPIの動作をすぐに確認できます。
ブラウザテスト画面はcurlと同じことをグラフィカルに行えるページで、
「Reactから見たらこんな感じになる」というイメージ確認にもなります。

> ⚠️ **これがないと…** バグに気づかないまま本番環境に出てしまいます。
> 「開発環境では動いていたのに本番で壊れた」を防ぐためにも動作確認は必須です。

### このフェーズでできるようになったこと

- `curl` でGET・POST・DELETE リクエストを送れる
- `Authorization: Bearer <token>` ヘッダーで認証付きリクエストを送れる
- ブラウザからAPIのログイン・一覧取得・追加・削除を操作できる
- HTTPステータスコードとレスポンスのJSONを読んで「正しく動いているか」判断できる

## まとめ：5つのフェーズがつながった

5つのフェーズを振り返ると、それぞれが前のフェーズの上に積み重なっていることがわかります。

```
フェーズ1：チームで安全に作業できる仕組みを整えた
    ↓
フェーズ2：データを保存・管理できる土台を作った
    ↓
フェーズ3：データを外部に渡す窓口（API）を作った
    ↓
フェーズ4：窓口の形を整えて、鍵（認証）をかけた
    ↓
フェーズ5：実際に動くか確かめた
```

どのフェーズも「次のフェーズのために必要だった」ということです。
順番が変わると成立しません。

### 次のステップ

このAPIはすでに **React などのフロントエンドから呼び出せる状態** です。
```javascript
fetch('/api/students', { headers: { Authorization: 'Bearer ' + token } })
```
と書くだけで、ブラウザ上でデータを取得・表示できます。

今後の学習では「このAPIを使って画面を作る」フロントエンド開発に進みます。
