# コマ25｜Users ⇔ Students をリレーションさせ、投稿者を表示する - 実装手順

## 今日のゴール

- `students` テーブルに「誰が登録したデータか」を表す `user_id` を持たせる
- `User` と `Student` をEloquentのリレーションでつなぐ（1人のユーザーが複数の生徒を持つ）
- 生徒一覧・詳細APIのレスポンスに「投稿者名（`posted_by`）」を含める
- 生徒を新規登録するとき、ログイン中のユーザーに自動で紐づけて保存する
- 一覧画面・詳細画面の両方に「投稿者」を表示する

> **位置づけ：** これまでの `students` テーブルは「誰が登録したか」の情報を持っていませんでした。`users` テーブルとの外部キーリレーションを追加し、その情報をAPI経由でフロントまで届けて画面に表示する、という一連の流れを作ります。

---

## 全体像：何を作るか

```
usersテーブル (1)
     │ user_id（外部キー）
     ▼
studentsテーブル (N)
     │ GET /api/students, GET /api/students/{id}
     │ レスポンスに posted_by（投稿者名）を含める
     ▼
StudentList.jsx（一覧）      StudentDetail.tsx（詳細）
     │ 投稿者列を追加              │ 投稿者項目を追加
     ▼                            ▼
一覧画面に投稿者名が表示される   詳細画面に投稿者名が表示される
```

> 💡 **たとえるなら…図書館の貸出カード**
>
> `students` テーブルの1件1件が「本」だとすると、`user_id` は「誰が借りた（登録した）本か」を書いた貸出カードです。カードが無いと、本を見ても「誰の担当か」が分かりません。まず本（`students`）にカードを挟む場所（`user_id` カラム）を作り、次に「1人の利用者は何冊も借りられる」という関係をコード（Eloquent）で表現し、最後にその情報を画面に出します。

Eloquentでは以下のように対応します。

| クラス | メソッド | 意味 |
|---|---|---|
| `User` | `students()` | 1人のユーザーは複数の生徒を持つ（`hasMany`） |
| `Student` | `user()` | 1件の生徒は1人のユーザーに属する（`belongsTo`） |

### なぜこの順番で作るか

バックエンドで `user_id` とリレーションを用意し、APIが `posted_by` を返せるようにならないと、フロントエンドは表示するデータそのものを持てません。そのため **バックエンド（マイグレーション → モデル → Controller/Seeder）→ フロントエンド（一覧 → 詳細）** の順に、小さく作って都度動かしながら進めます。

## Step 1｜バックエンド：マイグレーションで `user_id` を追加する

**🆕 新規ファイル** — 既存の `create_students_table` マイグレーションは直接編集しません。すでに一度実行済みのマイグレーションを書き換えるのは事故のもとなので、`user_id` カラムを追加する専用のマイグレーションを新しく作ります。

```bash
cd Laravel
php artisan make:migration add_user_id_to_students_table --table=students
```

`Laravel/database/migrations/` に、以下のような名前でファイルが生成されます（先頭は実行日時）。

```
2026_07_16_000001_add_user_id_to_students_table.php
```

生成されたファイルの中身を、以下のように書きます。

```php
<?php
// ▼ database/migrations/2026_07_16_000001_add_user_id_to_students_table.php

use Illuminate\Database\Migrations\Migration;
use Illuminate\Database\Schema\Blueprint;
use Illuminate\Support\Facades\Schema;

return new class extends Migration
{
    public function up(): void
    {
        // 既存の students テーブルに user_id カラムを追加する
        Schema::table('students', function (Blueprint $table) {
            $table->foreignId('user_id') // users.id を参照する外部キーカラムを作る
                ->after('id')             // id カラムの直後に配置する（見やすさのため）
                ->constrained()           // 参照先（users テーブルの id）を自動推測して外部キー制約を張る
                ->cascadeOnDelete();      // ユーザーが削除されたら、そのユーザーの生徒データも一緒に削除する
        });
    }

    public function down(): void
    {
        // migrate:rollback したときに、追加した外部キーとカラムを元に戻す
        Schema::table('students', function (Blueprint $table) {
            $table->dropForeign(['user_id']); // 先に外部キー制約を外す
            $table->dropColumn('user_id');    // カラム自体を削除する
        });
    }
};
```

- `Schema::table('students', ...)` … 既存テーブルに対して「カラムを追加する」変更なので、`create` ではなく `table` を使う
- `foreignId('user_id')` … `users.id` を参照する外部キーカラムを作る
- `->after('id')` … `id` の直後に `user_id` カラムを配置する（見やすさのため）
- `->constrained()` … 参照先を自動推測（`users` テーブルの `id`）して外部キー制約を張る
- `->cascadeOnDelete()` … ユーザーが削除されたら、そのユーザーが持つ生徒データも一緒に削除する
- `down()` … `migrate:rollback` したときに、追加した外部キーとカラムを元に戻せるようにしておく

> ⚠️ 途中から `user_id` を追加する場合、既存データには紐付け先がありません。開発中であれば `php artisan migrate:fresh --seed` でテーブルを作り直し、シーダーで最初からユーザー付きのデータを入れ直すのが簡単です（本番相当のデータを残したい場合は、一旦 `nullable()` で追加してからバックフィルし、後続のマイグレーションで `NOT NULL` 制約を付ける、という2段階の対応が必要になります）。

## Step 2｜バックエンド：モデルにリレーションを定義する

### 2-1. `app/Models/User.php`

**➕ 追加** — `HasMany` の `use` 文と `students()` メソッドを追加します。

```php
use Illuminate\Database\Eloquent\Relations\HasMany; // ← リレーションの型（HasMany）を使えるようにする

// ...

// 1人のユーザーは複数の生徒を持つ、という関係（1対多）を表すメソッド
// これを定義すると $user->students で「このユーザーが登録した生徒一覧」を取得できる
public function students(): HasMany
{
    // hasMany(Student::class) … 「Studentモデルをたくさん持っている」という意味
    return $this->hasMany(Student::class);
}
```

### 2-2. `app/Models/Student.php`

**➕ 追加** — `BelongsTo` の `use` 文と `user()` メソッドを追加します。

```php
use Illuminate\Database\Eloquent\Relations\BelongsTo; // ← リレーションの型（BelongsTo）を使えるようにする

// ...

// 1件の生徒は1人のユーザーに属する、という関係（多対1）を表すメソッド
// これを定義すると $student->user で「この生徒を登録したユーザー」を取得できる
public function user(): BelongsTo
{
    // belongsTo(User::class) … 「Userモデルに属している」という意味
    // ※ このテーブルの user_id カラムを見て、対応する users レコードを探しに行く
    return $this->belongsTo(User::class);
}
```

これで `$user->students` （複数の生徒を取得）と `$student->user` （その生徒の投稿者を取得）の両方向でアクセスできるようになります。

## Step 3｜バックエンド：Controllerで投稿者名を返す

### 3-1. `index()` を修正する

**🔁 置き換え** — `Student::all()` のような単純取得ではなく、`with('user')` でリレーション先を一緒に読み込み（Eager Loading）、`StudentResource` 経由でレスポンスに変換します。すでに `show()` が `StudentResource` を使っているので、`index()` も同じ形に揃えると、投稿者の出し方を1箇所（`StudentResource`）にまとめられます。

```php
public function index()
{
    // 生徒データを取得するときに、投稿者（user）の情報も一緒に読み込んでおく
    // → こうしておかないと、生徒の人数分だけ「投稿者名を取る」SQLが発行されてしまう（N+1問題）
    $students = Student::with('user')->get();

    // 取得した生徒データを、レスポンス用の形（StudentResource）に変換して返す
    return StudentResource::collection($students);
}
```

> 💡 **N+1問題とは？**
>
> `with('user')` を付けずに `Student::all()` や `Student::get()` と書くと、次のようなSQLが発行されます。
>
> 1. `students` テーブルから全生徒を取得するSQL（1回）
> 2. 画面やレスポンスで **生徒1件ごとに** `$student->user->name` にアクセスするたびに、「この生徒の投稿者（user）を1件取ってくる」SQL（生徒の人数分＝N回）
>
> つまり、生徒が10人いれば「生徒一覧を取る1回」＋「投稿者を取る10回」＝ **合計11回** のSQLが発行されてしまいます。生徒が100人・1000人と増えるほど、このSQLの回数も比例して増えていき、APIがどんどん遅くなります。これが **N+1問題**（1回で済むはずが、N件のデータに対してN回の追加クエリが走ってしまう問題）です。
>
> `with('user')` を先頭に付けておくと、Laravelは
>
> 1. `students` テーブルを取得するSQL（1回）
> 2. そこに登場した `user_id` をまとめて、対応する `users` を一括取得するSQL（1回）
>
> の **合計2回** で済ませてくれます（これを **Eager Loading**＝事前に一括読み込み、と呼びます）。生徒が何人に増えても、SQLの発行回数は常に2回のままです。「後から1件ずつ取りに行く（N+1）」のではなく「最初にまとめて取っておく（Eager Loading）」のがポイントです。
>
> ⚠️ 戻り値の型を `: JsonResponse` と書きたくなりますが、この場合は `use Illuminate\Http\JsonResponse;` のimportが必要です。他の `store()`/`show()`/`update()`/`destroy()` も型を書いていないので、揃えて型なしのままにしておきます。

### 3-2. `app/Http/Resources/StudentResource.php` に `posted_by` を追加する

**➕ 追加** — `index()` を直しただけでは、レスポンスに `posted_by` はまだ出てきません。実際にJSONへ変換しているのは `StudentResource` なので、ここに追加します。`show()`（詳細API）も同じ `StudentResource` を通るので、1回の修正で一覧・詳細の両方に反映されます。

```php
public function toArray($request): array
{
    // この生徒データを登録した投稿者（ユーザー）の名前を、先に変数に取り出しておく
    $postedByName = $this->user->name;

    return [
        'id'         => $this->id,
        'name'       => $this->name,
        'email'      => $this->email,
        'score'      => $this->score,
        'evaluation' => $this->score >= 80 ? 'A' : 'B',
        'created_at' => $this->created_at->format('Y/m/d'),
        'posted_by'  => $postedByName, // ★ 追加：投稿者の名前
    ];
}
```

### 3-3. `store()` を修正する

**🔁 置き換え** — `Student::create()` ではなく、ログイン中のユーザー経由で作成することで、`user_id` が自動でセットされるようにします。バリデーションは既存のまま `$request->validate(...)` を使い、その結果（`$validated`）をそのまま生徒作成に渡します。

```php
public function store(Request $request)
{
    // バリデーション：送られてきたデータの形式を確認する
    $validated = $request->validate([
        'name'  => 'required|max:100',          // 必須・100文字以内
        'email' => 'required|email|unique:students', // 必須・メール形式・重複不可
        'score' => 'required|integer|min:0|max:100', // 必須・整数・0〜100
    ]);

    // $request->user() … 今ログインしているユーザー（Sanctumが認証してくれている）
    // ->students()->create(...) … そのユーザーの生徒として新規作成する
    //   → Eloquentが自動で user_id にログインユーザーのIDを入れてくれる
    $student = $request->user()->students()->create($validated);

    // 201 Created（作成成功）のステータスコードで返す
    return response()->json([
        'status' => 'ok',
        'data'   => $student,
    ], 201);
}
```

`$request->user()` はSanctumで認証済みのユーザー（＝今ログインしている講師）です。`->students()->create(...)` と書くことで、Eloquentが自動的に `user_id` にそのユーザーのIDを埋めてくれます。`Student::create($request->all())` のように「全部のパラメータ」を渡すのではなく、バリデーション済みの `$validated` だけを渡す点にも注目してください。

## Step 4｜バックエンド：シーダーもユーザー経由に変更する

このプロジェクトには `StudentSeeder.php` というファイルはなく、`database/seeders/DatabaseSeeder.php` の中で直接 `Student::factory(10)->create()` として生徒データを作っています。

```php
// database/seeders/DatabaseSeeder.php（今の中身）
public function run(): void
{
    \App\Models\Student::factory(10)->create(); // 生徒データを10件作成
}
```

このままだと生徒データに `user_id` が入らない（そもそも紐づけ先のユーザーがいない）ので、**①テストユーザーを先に作る → ②そのユーザー経由で生徒を作る** という2段階に書き換えます。新しいファイルを作る必要はなく、`DatabaseSeeder.php` を直接直すだけでOKです。

**🔁 置き換え**

```php
// database/seeders/DatabaseSeeder.php（修正後）
public function run(): void
{
    // ①ログイン用のテストユーザーを1人作る
    // User::factory() … UserFactory に定義されたダミーデータで作成する
    //   password は UserFactory 側ですでに「password」をハッシュ化した値になっているので、
    //   ここで name と email だけ上書きすれば test@example.com / password でログインできる
    $user = \App\Models\User::factory()->create([
        'name'  => 'テストユーザー',
        'email' => 'test@example.com',
    ]);

    // ②生徒データを作るときに ->for($user) を挟むと、
    //   Student::user() のリレーション（belongsTo）を通じて、自動で user_id に $user->id が入る
    \App\Models\Student::factory(10)->for($user)->create();
}
```

`Student::factory(10)->for($user)` の `for()` は「このモデル（$user）に関連付けて作る」という意味のFactoryの機能です。`Student` モデルには `user()` （`belongsTo(User::class)`）が定義済みなので、Laravelが自動的に `user_id` を `$user->id` に設定してくれます。`$user->students()->create(...)` を1件ずつ呼ぶのと同じ結果を、Factoryでまとめて実現しているイメージです。

## まとめ

| 概念 | 今回の対応箇所 |
|---|---|
| 1対多リレーション | `students.user_id`、`User::students()`、`Student::user()` |
| Eager Loading（N+1対策） | `StudentController@index` の `Student::with('user')` |
| ログインユーザーとの自動紐付け | `StudentController@store` の `$request->user()->students()->create(...)` |
| APIレスポンスへの投稿者情報追加 | `posted_by` フィールド |
| フロントへの表示 | `StudentList.tsx`／`StudentDetail.tsx` |
